# Cyber-Eden Pitfalls (前向避坑指南)

## [架构] 防止自身内存泄漏
- **陷阱**：无尽日志 `Diary.txt` 如果不作处理并且完全无截断附加给每一次祈祷（Prompt），将伴随沙盒时间演进立刻流失突破 Token 上限。
- **教训**：预留空间并在早期观察，如果 Token 增长迅速，考虑上帝微服务下发强制指示或者在 Prompt 中诱导 Adam 发明文件摘要读写系统。

## [安全与架构边界] Docker 通信管道保护
- **陷阱**：Oracle Proxy 与 Adam Agent 的通讯通道如果通过开放普通的 Docker host network，存在暴露整个宿主内网环境漏洞或者让 Adam 窃得绕包能力隐患。
- **教训**：Agent 与神谕祭坛之间绝不传递敏感 Key 密文。仅沙盒必须将自身与外网隔离，只允许 `host.docker.internal:8000` 放行请求大模型，以此严格杜绝 Agent 跳出沙盒破坏机器。

## [数据原子性] 自我覆写过程的致命性
- **陷阱**：Agent 使用 `open('agent.py', 'w')` 覆写自己是非常脆弱的。如果写入中途 Docker crashed 或系统资源耗尽，旧代码将直接清空而新代码也是乱码。
- **教训**：**必须引入备份机制或写入重定向！** 在 Archangel 这边要在内存/物理磁盘上随时保持上一份哈希安全的代码的完整快照副本（Snapshot Backup）。即使 Agent 写烂了自己，大天使依然可以直接强行回滚抢救回来。

## [权限] 圣言不可篡改机制 (Bible Isolation)
- **陷阱**：既然我们将沙盒挂载为一个文件夹，如果 `Bible.txt` 用 rw 挂载，只要 Agent 发疯或者在代码层面使用全覆盖 `*.*` 逻辑，上帝制定的唯一高维规则 `Bible` 会被立刻摧毁。
- **教训**：必须在 Host 端挂载目录启动时，将 `Bible.txt` 作为单文件的只读（`:ro`）卷挂载进容器，在操纵系统最底层剥夺 Agent 修改物理真理的可能。
